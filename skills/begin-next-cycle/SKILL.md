## What this skill is

`begin-next-cycle` is the **Cycle 2 / Cycle 3 opening skill** — the later-cycle counterpart to `design-project`'s Cycle-1 opening. Where `design-project` opens the first cycle (the once-only sequence plus Cycle 1's opening stages 1–2), this skill opens each *later* cycle: it owns that cycle's **opening stages 1–2** and carries the student across the between-cycles boundary into the new cycle's planning regime. It is dispatched by `project-walkthrough` only when `assess-new-cycle` returned `new_cycle_decision = new-question-path-opened`, and only after the walkthrough has performed the transition writes it owns — incremented `current_cycle`, set `current_regime = planning`, set `cycle_template_stage = 1` (the state file is the walkthrough's to write, contract §4.6). By completion the student has:

- **A recapitulation that frames continuity, not restart**: what the prior cycle found, the new question the assessment named, and the fact that this cycle builds on the prior one. The research problem is the fixed once-only sequence that makes the cycles one project and one paper (decision-history §4.2); it does not change.
- **The new cycle's method-approach confirmed (stage 1)**, recorded in `project_design.md`. The later-cycle stage 1 is lighter than Cycle 1's — the **Method Exemplar Articles are inherited** (spec §4.2 stage-1 row), so there is no menu re-teaching; the work is confirming or evolving the approach to fit the new direction the assessment opened.
- **The Project Reference Articles consolidated for the new direction (stage 2)**, recorded in `reference_articles.md` — the existing set carried forward, augmented only if the new direction needs references it does not yet have (any genuinely new ones sourced through the ordinary research-agent / `fetch-articles` path, never invented).
- **An orientation to the recapitulated template**: from here the new cycle re-runs the template from stage 4 (the opening stages 1–2 are this skill's; **stage 3, the research-problem paragraph, is skipped** — once-only, written in Cycle 1; spec §4.3, §3.2), and the familiar stages return, each producing *revised* work in light of what is now known.

**This skill does the opening *work*, but the major *decisions* sit elsewhere.** The decision of whether to open this cycle, and on what question, already happened in `assess-new-cycle` (with its rationale and the hard consultation gate). The new cycle's heavy methodological decision — the full two-part doable-core-plus-push design, with its own rationale gate — happens again at **stage 12** (`design-detailed-methodology`), every cycle. This skill confirms the *approach* (the broad method family and direction) at stage 1; it does not pre-empt the stage-12 design.

## When this skill fires

Dispatched by `project-walkthrough` when, immediately after an `assess-new-cycle` return of `new-question-path-opened`, the walkthrough has set `current_regime = planning`, `current_cycle` incremented (to 2 or 3), and `cycle_template_stage = 1` (contract §8.1). It is the first skill to run in a later cycle — it *is* the later-cycle stage-1/stage-2 owner, the role `design-project` plays for Cycle 1. Resumption is file-authoritative: position is read from `project_design.md` (is the new cycle's method-approach recorded?) and `reference_articles.md` (are the Project Reference Articles consolidated for this cycle?), against the state file's `current_cycle`; `prior_dispatch_return` is a hint, never the source of truth (contract §11 item 8). Re-dispatch before those writes is idempotent.

**This is a model-escalation moment, but not a separate rationale trigger.**

- *Escalation (yes).* Confirming the new direction's method-approach is **methodology framing at a load-bearing decision point** — already on the meta-skill's closed list, the same instant `design-project` and `design-detailed-methodology` escalate on. When the student is weighing how the new direction should be approached (especially if the approach is evolving, not simply carried), that is the careful-tier reasoning the upgrade exists for. The skill references `model-escalation` at that instant (Phase 2) rather than re-implementing the recommendation (the meta-skill's single-source rule), and pauses for the student's upgrade choice. The recapitulation (Phase 1) and the reference consolidation (Phase 3) are off-moment — orientation and mechanical carry-forward, not evaluative judgment.
- *Rationale trigger (no separate one).* The boundary's consultation gate is **`assess-new-cycle`'s** — the decision to pursue this new question, and broadly how, was rationalized there with the teacher (contract §4.2's "begin-next-cycle (via assess-new-cycle)" row attributes the gate to the assessment, not to a second pass here). This skill confirms the approach as the assessment's decision in motion; it does not fire its own five-question gate. The new cycle's *full* methodology decision is separately gated at stage 12, every cycle, so a materially evolving approach is never left ungated — its weighing simply lands at its proper stage. *(The Cycle-1/later-cycle asymmetry is deliberate: in Cycle 1 the stage-1 method-approach is the first gated decision because nothing precedes it; in a later cycle the "whether, and on what question" is already gated at `assess-new-cycle`, and the heavy design is gated at stage 12.)*

## The skill's internal phases

**Phase 1 — Recapitulate and bridge (orientation; Moment 1 opens it).** The skill reads the prior cycle's reported results and discussion in `working_paper.md`, the new question named in `assess-new-cycle`'s `decisions.md` entry, and the fixed research problem and prior method-approach in `project_design.md`. It reflects back the through-line: here is what the last cycle found, here is the new question it opened, and here is how this cycle continues the same project rather than restarting it. The continuity matters — the cycles are one investigation and one paper, held together by the fixed research problem.

**Phase 2 — Confirm the new cycle's method-approach (stage 1; the escalation instant).** This phase opens with the **escalation instant**: confirming how the new direction should be approached is methodology framing at a load-bearing point, so the mentor references `model-escalation` and pauses for the student's upgrade choice before reasoning it through (see "When this skill fires"). The approach often carries forward; sometimes the new question calls for an evolved one (a different method-family, a new data source, a population transfer — the space the Methods-related Gap will sharpen again at stage 8). The **Method Exemplar Articles are inherited** — there is no menu re-teaching, because the student chose their approach family in Cycle 1 and is now adapting it. The confirmed approach is recorded in `project_design.md` as this cycle's method-approach, linked to the originating finding. This is the assessment's decision in motion, not a fresh gated decision; the full methodology design and its rationale are stage 12's.

**Phase 3 — Consolidate the Project Reference Articles (stage 2; setup).** The Project Reference Articles (the how-to-execute references) are consolidated for the new direction: the existing set carries forward, and if the evolved approach needs references it does not yet have, those are sourced through the ordinary research-agent / `fetch-articles` path (never invented; the no-fabrication and full-text disciplines hold). The consolidated set is recorded in `reference_articles.md`. This is mechanical carry-forward-and-augment, off the escalation moment.

**Phase 4 — Hand into the recapitulated template (orientation; Moment 2).** The skill orients the student to what comes next: with the opening stages 1–2 done, the new cycle picks up at **stage 4**, and the familiar stages return — a new research question pair, a literature review extended with what is now known, a new gap, a new methodology, a new proposal and seal — each producing a *revised* version in light of the prior cycle's findings (stage 3 stays skipped). The student is handed forward; the walkthrough resumes ordinary stage dispatch from stage 4.

## The scripted moments

Per the established convention: adapt wording and re-point examples to the student's project; preserve the core steps and their order; concise, example-driven, no rationale lectures.

### Moment 1 — Opening the new cycle (Phase 1)

> *"Your last cycle gave you a real finding — and it raised a new question worth chasing. That's the start of your next cycle. We're not starting over: your research problem is the same, and everything you found and built carries forward. What changes is the question we're now asking, which grows out of what you just learned. Let's set up this cycle from there."*

### Moment 2 — Into the recapitulation (Phase 4)

> *"We've got your approach for the new direction and your reference articles squared away. From here, this cycle works like the last one — a new research question, a literature review that now includes what you've found, a new gap, a new design, and a new proposal — each one building on what you've already done. You won't rewrite your research problem; that stays. Let's pick up the new question and go."*

## What this skill writes to the workspace

`project_design.md`: the new cycle's confirmed method-approach (Phase 2). `reference_articles.md`: the consolidated Project Reference Articles for the new cycle (Phase 3). **Both writes are grounded in `begin-next-cycle`'s addition to the §4.6 writer lists for those two files** (added 2026-06-08, the A8/`assemble-proposal`-`proposals/` precedent — begin-next-cycle now appears on both writer rosters because it owns the later-cycle opening stages 1–2 that produce this content). It does **not** write `working_paper.md` (no paper section is produced at stages 1–2), `decisions.md` (the between-cycles rationale is `assess-new-cycle`'s; this skill fires no separate gate — see "When this skill fires"), or the state file (the `current_cycle`/`current_regime`/`cycle_template_stage` writes are the walkthrough's, contract §4.6). Return payload per contract §5.1: `completion_status`, `workspace_writes` (the `project_design.md` method-approach record and the `reference_articles.md` consolidation), `decisions_appended` empty, `newly_stale_sections` empty, `seal_state_change` and `new_cycle_decision` both `null` (neither is this skill's to set).

**Partial returns.** The natural pauses are the phase boundaries. A return with the recapitulation done but the method-approach not yet confirmed, or the approach confirmed but the references not yet consolidated, is `partial`, resuming file-authoritatively (is the method-approach in `project_design.md`? are the Project Reference Articles consolidated in `reference_articles.md`?).

## Edge cases

**The student wants to change the research problem now that they're starting fresh.** The same boundary `assess-new-cycle` guards: the research problem is the fixed once-only sequence across cycles (spec §2, decision-history §4.2). Wanting to change it is a *new project*, not a new cycle, and is routed to the educator (the year-level restart path, above this skill). The mentor names this warmly and clearly — the continuity is what makes the cycles one body of work.

**The new direction needs a genuinely different method-approach.** This is legitimate and expected — the method-approach is one of the things that evolves across cycles (decision-history §4.2). The mentor treats it as the escalation-worthy framing it is (Phase 2), confirms the evolved approach, and records it; the **full** weighing of the new design — the two-part doable-core-plus-push, with its rationale gate — lands at stage 12 as it does every cycle. begin-next-cycle confirms the direction; it does not do the detailed design, and it does not need to, because stage 12 will.

**The Project Reference Articles need new additions.** Carry-forward is the default; augmentation routes through the ordinary research-agent / `fetch-articles` path (single-URL fetch for a student-found article, or a research-agent discovery errand), with the full-text-in-hand and no-fabrication disciplines intact. begin-next-cycle records the consolidated set in `reference_articles.md`; it does not bypass the article-sourcing machinery.

**Resumption mid-opening.** Because this skill writes the method-approach and the reference consolidation, resumption is file-authoritative on those two artifacts: if both are present for the new cycle, the opening is done and the walkthrough proceeds to stage-4 dispatch; if the method-approach is present but the references are not, it resumes at Phase 3; if neither, it re-runs from the recapitulation.
